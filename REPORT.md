<p> Лабораторная работа №12 </p>

<h1>Цель работы:</h1> <h2> Освоить работу с Git и GitHub: создание локального репозитория, фиксация изменений, работа с ветками, слияние, а также размещение статического и динамического сайта на GitHub.   </h2>

<h1>Выводы и документация:</h1>

* <h2>Процесс выполнения:</h2>
Для выполнения работы была установлен Git, после чего в выбранной (/C/project/lab12) директории был создан новый локальный репозиторий. В каталоге был подготовлен файл REPORT.md, в который было занесено описание выполнения лабораторной работы. Содержимое файла было добавлено в репозиторий и зафиксировано. Для дальнейшего оформления была создана отдельная ветка, предназначенная для отчёта. В этой ветке отчет был дополнен, после чего изменения были сохранены. По завершении редактирования ветка была объединена с основной (master), и актуальная версия отчёта стала частью главной ветки проекта.

  После подготовки локального репозитория был создан удалённый на GitHub с названием «homework_IIT». Связь локального и удалённого репозиториев была настроена, и содержимое было отправлено на сервер, в удаленный репозиторий. Далее был настроен GitHub Pages, благодаря чему сайт стал доступен как веб-страница. В рамках дополнительного задания был также размещён динамический сайт, выполненный ранее на WordPress. Его структура была добавлена (вместе со всеми файлами) в виде исходных файлов, так как GitHub Pages не поддерживает серверный PHP-функционал.



* <h2>Трудности:</h2>
Наибольшие сложности при выполнении лабораторной работы возникли при установке Git и настройке связки с GitHub. Во-первых, при первой инициализации репозитория Git не видел имя пользователя и email, поэтому пришлось выполнить комнады: git config --global user.name и git config --global user.email.
 Во-вторых, при попытке залить проект на GitHub появлялись ошибки аутентификации,при выполнении команды push, так как GitHub прекратил поддержку входа в систему с использованием учетной записи (логина) и обычного пароля при работе через командную строку (что было сделано для повышения безопасности), поэтому была настроена авторизация через GitHub Personal Access Token. Также вызвало затруднение при выполнеии команды pull, так как возникада ошибка, связанная с тем, что размер файла сайтом Wordpress по объему был слишком большим. Поэтому пришлось через pip устанавливать git-filter-repo, чтобы удалить все большие (>100МБ) файлы из истории и установить lfs, чтобы при повторном выполнении push можно было загрузить уже все файлы. Отдельно пришлось разобраться, как правильно разместить статический сайт через GitHub Pages — нужно было выбрать ветку, указать путь, где находились файлы с сайтом, (изначально файлы находились в отдельной папке в репозитории, но при попытке указать путь к этой папке, из которой должен был развертываться сайт, оказалось, что GitHub не видит путь к папке, поэтому пришлось переместить все файлы, связанные с сайтом, в корень) и дождаться активации публикации.


* <h2>Вывод:</h2>
Выполнение данной лабораторной работы позволило погрузиться в принципы работы с системой Git и GitHub и понять, насколько важным является грамотное управление изменениями в любом проекте — даже самом небольшом. Если раньше процесс создания файлов, их хранения и редактирования воспринимался как нечто локальное и ограниченное одним устройством, то работа с Git и GitHub показала, насколько по-раному может быть организован процесс работы, когда каждая правка становится отслеживаемой и каждая версия сохраняется. Постепенное выполнение лабораторной работы — начиная с установки Git и завершив публикацией сайтов на GitHub — наглядно продемонстрировало, насколько важно уметь работать как с локальным, так и с удалённым репозиториями.

  В процессе выполнения лабораторной работы стало очевидно, что Git — это не просто инструмент для сохранения файлов, а полноценная система, которая требует аккуратности, последовательности и понимания основных концепций: коммитов, веток, слияний, локальных и удалённых репозиториев. Особое значение приобрела работа с ветками, поскольку именно она показала, как можно безопасно экспериментировать, дополнять проект, не нарушая его рабочую версию. Публикация статического сайта на GitHub Pages дала практическое понимание того, как исходный код превращается в реальный сайт доступный всем пользователям. Параллельно стало очевидно, что не все типы проектов одинаково совместимы с GitHub Pages, что проявилось при попытке разместить динамический сайт. Это дало полезный опыт понимания ограничений хостингов.
